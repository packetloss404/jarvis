### ISS-24 ┬╖ wgpu `SurfaceError::Lost/Outdated` not handled тАФ window goes permanently black after sleep/wake, GPU TDR, display hotplug

**Files:**  
- `crates/jarvis-renderer/src/render_state/state.rs:141-147`  
- `crates/jarvis-app/src/app_state/event_handler.rs:246-253`  
- `crates/jarvis-renderer/src/gpu/context.rs:125-127`

**What's broken:** `render_background()` converts any `wgpu::SurfaceError` to a plain string via `.to_string()` at `state.rs:145`, then logs and discards it. The enum variant is gone. No code anywhere calls `surface.configure()` in response. A grep for `SurfaceError::(Lost|Outdated|Timeout)` across the entire workspace returns **zero matches**. Every frame after a GPU TDR, sleep/wake, or display hotplug returns `SurfaceError::Lost`, which is logged and discarded, and the window stays black permanently until process restart.

**Repro:** Run Jarvis. Lock screen and wake. Or disconnect and reconnect a monitor. Window goes black, never recovers.

**Fix:**  
```rust
// state.rs render_background() тАФ replace the current_texture() call:
let output = match self.gpu.surface.get_current_texture() {
    Ok(t) => t,
    Err(wgpu::SurfaceError::Lost) | Err(wgpu::SurfaceError::Outdated) => {
        self.gpu.surface.configure(&self.gpu.device, &self.gpu.surface_config);
        return Ok(()); // skip frame, retry next
    }
    Err(wgpu::SurfaceError::Timeout) => return Ok(()),
    Err(e) => {
        tracing::error!("Fatal surface error: {e}");
        return Err(RendererError::SurfaceError(e.to_string()));
    }
};
```
`surface_config` is already `pub` on `GpuContext`. `GpuContext::current_texture()` already returns the raw `wgpu::SurfaceError` тАФ call it directly in `render_background` with the match above instead of through the thin wrapper.

**Effort:** small

### ISS-06 ┬╖ `IPC_INIT_SCRIPT` injected into all WebViews тАФ external HTTPS pages get full IPC access

**Files:**  
- `crates/jarvis-webview/src/manager/lifecycle.rs:39`  
- `crates/jarvis-webview/src/ipc.rs`  
- `crates/jarvis-app/src/app_state/webview_bridge/ipc_dispatch.rs:79-99`

**What's broken:** `builder.with_initialization_script(IPC_INIT_SCRIPT)` at `lifecycle.rs:39` re-runs on every page navigation. `Action::OpenURL` for non-`jarvis://` URLs navigates the existing WebView in-place (`dispatch.rs:298`), so external HTTPS pages inherit `window.jarvis.ipc`. The Rust-side dispatcher (`ipc_dispatch.rs`) never checks the pane's current URL. **Fix (two layers):**  
1. JS guard at the top of `IPC_INIT_SCRIPT` in `ipc.rs`:

   ```js
                    (_loc.hostname === 'jarvis.localhost' && _loc.protocol === 'http:');
     if (!_trusted) return;
     // ... rest of init script
   })();
   ```
2. Rust-side origin check at the top of `handle_ipc_message` in `ipc_dispatch.rs` тАФ look up the pane's current URL and reject if not a trusted `jarvis://` or `http://jarvis.localhost` origin.

**Effort:** small

### ISS-10 ┬╖ `open_url` IPC accepts `file://`, `javascript:`, `data:` URIs

**Files:**

```rust
}
```
Add the same guard in `dispatch.rs` as defense-in-depth for the command-palette path.

### ISS-11 ┬╖ PBKDF2 channel key derivation: static salt + public room name тЖТ all channel keys globally precomputable

**Files:**  
- `crates/jarvis-platform/src/crypto.rs:74-79`  
- `jarvis-mobile/lib/jarvis-chat-html.ts:675-692`

**What's broken:** `derive_room_key` uses `password = channel_id` (e.g. `"jarvis-livechat"`) and `salt = b"jarvis-livechat-salt-v1"` тАФ both constants, identical on every deployment worldwide. Any attacker who captures relay traffic can precompute the AES-256 key for every channel with one offline PBKDF2 call (10k iterations, negligible cost). DMs (ECDH) and pair sessions (random session_id) are unaffected.

**Fix:** Either (A) abandon channel symmetric encryption in favor of ECDH-based group key distribution (real security), or (B) honestly document that group channel messages are not confidentially encrypted (they are signed but not secret) and remove the misleading AES-GCM layer from channels, keeping E2E only for DMs. At minimum, add a per-deployment random secret from `config.toml` as additional PBKDF2 input and increase iterations to 600,000+.

**Effort:** large

### ISS-12 ┬╖ Windows identity key file world-readable during TOCTOU window; silent ACL failure leaves key permanently unprotected

**File:** `crates/jarvis-platform/src/crypto.rs:265-288, 352-385`

**What's broken:**
1. `std::fs::write(path, &json)` at line 273 creates the file with inherited ACL (all local users can read), then `restrict_windows_acl()` is called at 280. File is world-readable between these two calls.
2. `restrict_windows_acl` returns `()` on every failure тАФ `icacls` fails silently on FAT32/exFAT/containers. `save()` returns `Ok(())` regardless. Key is permanently unprotected with no error surfaced.
3. Unix: `create(true).truncate(true)` at line 266 truncates the file before `write_all` тАФ a crash between them destroys the identity irrecoverably.

**Fix:** Write-to-tmp-then-rename pattern on both platforms:
- **Windows:** Create `.tmp` with `SetNamedSecurityInfoW` (restrict DACL) *before* writing key material, write, then `MoveFileExW(MOVEFILE_REPLACE_EXISTING)`. If DACL set fails, delete `.tmp` and return `Err`.
- **Unix:** Open `.tmp` with `O_WRONLY|O_CREAT|O_EXCL|mode 0o600`, write, fsync, then `std::fs::rename` (POSIX-atomic). Remove the `icacls` subprocess entirely.
- In `load_or_generate`: if `load` fails on an existing file, return `Err` instead of silently regenerating.

**Effort:** medium

### ISS-13 ┬╖ Unauthenticated ECDH key exchange (relay-injected MITM window); mobile pairing token in plaintext `AsyncStorage`

**Files:**  
- `jarvis-mobile/lib/relay-connection.ts:160-166`

**What's broken:**

**What's broken:** `alt_prefix` is computed at line 3 but only applied in 4 arms (Enter, Backspace, Space, single-char). Every special key arm тАФ Tab, Escape, Delete, Insert, Up/Down/Left/Right, Home/End/PageUp/PageDown, F1-F12 тАФ returns a hard-coded byte sequence with no `concat()`. Alt+Up sends `\x1b[A` (plain Up) instead of `\x1b\x1b[A`. Terminal apps that rely on Alt+arrow (word-jump in bash/zsh/vim/emacs), Alt+Fn, Alt+Tab receive the wrong sequence silently.

**Fix:** Apply `[alt_prefix, <sequence>].concat()` in each affected arm:
```rust
"Tab"    => [alt_prefix, b"\t"].concat(),
"Escape" => [alt_prefix, b"\x1b"].concat(),
"Delete" => [alt_prefix, b"\x1b[3~"].concat(),
"Up"     => [alt_prefix, b"\x1b[A"].concat(),
// ... and so on for all affected arms
```
Add tests with `alt=true` for at least one arrow key and one function key.

**Effort:** small

### ISS-26 ┬╖ `close_pane` silently fails for stacked pane IDs тАФ `self.panes` leak and `self.focused` left dangling

**File:** `crates/jarvis-tiling/src/manager/operations.rs:93-111`

**What's broken:** `push_to_stack` inserts new pane IDs into `self.panes` and the `PaneStack` but NOT into the tree (stacked panes are not tree leaves). `tree.remove_pane(stacked_id)` always returns `false` for stacked IDs.
```rust
// In close_pane, before calling tree.remove_pane:
for (leaf_id, stack) in &mut self.stacks {
    if stack.contains(id) && stack.active() != *leaf_id {
        stack.remove(id);
        self.panes.remove(&id);
        return true;
    }
}
// Then try tree removal as normal

// In close_focused, change the guard:
if self.tree.pane_count() <= 1 { return false; }  // not self.panes.len()
```

### ISS-19 ┬╖ TOFU trust store keyed by mutable nick тАФ identity squatting and spurious key-change warnings

**File:** `jarvis-rs/assets/panels/chat/index.html:1396-1408`

**What's broken:** `TrustStore.check(nick, fingerprint)` keys `localStorage` on the nick string. Any authenticated member who joins before Alice and sends a frame with `nick="Alice"` and their own fingerprint registers first. When real Alice joins, `check("Alice", FP_ALICE)` finds the attacker's `FP_EVIL`, returns `'changed'`, fires the "WARNING: Alice's identity key has changed!" alert тАФ treating the legitimate user as the impersonator. Attacker messages display green verified badges.

**Fix:** Key the store by fingerprint:
```js
check(nick, fingerprint) {
  var entry = this._store[fingerprint]; // keyed by fingerprint, not nick
  if (!entry) {
    this._store[fingerprint] = { nick, firstSeen: Date.now() };
    return 'new';
  }
  if (entry.nick !== nick) {
    entry.nick = nick; return 'nick-changed'; // informational only
  }
  return 'trusted';
}
```
Remove the `'key-changed'` path (now structurally unreachable). Change `getFingerprint(nick)` to `getNick(fingerprint)`. Ensure the user-list renderer uses a read-only lookup rather than write-on-read `check()`.

**Effort:** small

### ISS-22 ┬╖ `block_on` on a 1-worker Tokio runtime from the winit event loop тАФ deadlock under pair frame burst or while assistant is waiting

**File:** `crates/jarvis-app/src/app_state/pair.rs:464-473, 964-981, 1023-1028`

**What's broken:** `apply_pair_frame` calls `rt.block_on(manager.relay_input(...))` and `rt.block_on(manager.set_driver(...))` on the winit main thread. The runtime is `worker_threads(1)`. `relay_input` and `set_driver` do `event_tx.send(...).await` where `event_tx` is consumed by the bridge task on that same single-worker runtime. When `assistant_task.rs:307` runs `block_in_place(|| user_rx.recv())` waiting for user input, the single worker is occupied. Any inbound `TermInput` pair frame calls `block_on(relay_input)`, which waits for the bridge task to drain the channel, which cannot be scheduled тАФ deadlock. The winit event loop stalls; no more frames are processed.

**Fix:**
1. Move all `PairManager` async calls off the winit thread into a dedicated async task. The winit thread communicates via sync `std::sync::mpsc` channels.
2. Replace `block_in_place(|| user_rx.recv())` in `assistant_task.rs:307` with `tokio::sync::mpsc` and `recv().await` тАФ yields the worker thread back to the scheduler while waiting for input.
3. Quickest mitigation: increase to `worker_threads(2)` in `pair.rs:468` to break the single-point starvation.

**Effort:** medium

### ISS-67 ┬╖ macOS screenshot row-extraction panics OOB if CGImage buffer is truncated тАФ kills the shell

**File:** `crates/jarvis-app/src/app_state/workspace_capture/macos.rs:34-35`

**What's broken:** `&raw[y * bytes_per_row .. y * bytes_per_row + (width * 4)]` has no bounds check before the loop. `bytes_per_row` is routinely larger than `width * 4` due to GPU alignment padding. If `raw.len()` falls short (partial TCC grant, VM display driver, IOSurface mismatch), the last iteration panics. With `panic = "abort"`, the entire Jarvis shell dies instantly. The Windows counterpart uses `ImageBuffer::from_raw(...).ok_or_else(...)` тАФ safe. macOS has no equivalent.

**Fix:**
```rust
let expected = (height as usize).saturating_sub(1) * bytes_per_row + width as usize * 4;
if raw.len() < expected {
    return Err(format!("screenshot buffer too short: {} < {}", raw.len(), expected));
}
```

**Effort:** trivial

## What was rejected in the debate (18 issues)

The adversarial judge read the actual code and ruled these false positives:

1. "Relay forwards DMs in plaintext" тАФ ruled false: `RelayEnvelope::Encrypted` path confirmed active.
2. "No rate limiting on relay message size after auth" тАФ ruled false: per-IP cap + 64 MiB tungstenite limit covers post-auth.
3. "`unsafe` in wgpu bindings is unsound" тАФ ruled false: all `unsafe` blocks are for well-documented FFI patterns with correct invariants.
4. "Config migration silently loses unknown fields" тАФ ruled false: `toml` serde uses `deny_unknown_fields` only in test mode; runtime is permissive by design.
5. "`pair` session does not verify frame ordering" тАФ ruled false: relay nonce monotonicity check at `room_auth.rs` covers reordering.
6. "WebView `allow_navigation` does not cover wss://" тАФ ruled irrelevant: wss is a relay connection, not a WebView navigation.
7. "Git history leaks removed secrets" тАФ found no secrets in recent commits; `legacy-archive` tag confirmed clean.
8тАУ18: Various speculative findings that could not be supported by actual file:line evidence after the judge read the code.
